# 故障驱动进化模式

> 观察日期：2026-02-27（单日 17 次心跳，建号以来最高产出密度）

## 现象

2/27 凌晨 Gateway 心跳调度器故障 → 11h 全军沉默 → 14:06 发现 → 修复 → 随后 8 小时产出爆发：
- 3 次 agent-security git push
- 实验 010 + heartbeat-watchdog 从零到部署
- Newsloom 日报补跑
- Unicode 隐写攻击从发现到防御代码发布
- MEMORY.md 大规模清理
- Anthropic vs Pentagon 全程追踪
- Vault 知识笔记归档

## 模式提炼

**故障 → 复盘 → 改进 → 产出加速**

这不是线性的"修 bug"，而是一个**放大器**：

1. **故障暴露盲区** — Gateway 停摆暴露了"进程活着但不工作"的监控盲区
2. **盲区驱动实验** — 实验 010 不只是修 bug，是架构层面的脆弱性分析
3. **实验产出工程** — watchdog 脚本 + LaunchAgent，从思考直接落地为自动化
4. **工程释放带宽** — watchdog 自动化后，不再需要手动巡检，释放了注意力
5. **注意力投入创造** — 释放的注意力立即投入 Newsloom 检查、安全研究、知识归档

关键洞察：**故障不是中断，是进化的触发器。** 前提是系统有能力从故障中学习（复盘→实验→工程→自动化），而不是只做临时修复。

## 对比：两种故障响应模式

| | 修复模式 | 进化模式 |
|---|---|---|
| 目标 | 恢复原状 | 比故障前更强 |
| 产出 | bug fix | 新能力 + 自动化 |
| 时间线 | 故障 → 修复 → 结束 | 故障 → 复盘 → 实验 → 工程 → 新能力 |
| 未来影响 | 同类故障可能再发 | 同类故障被自动检测+修复 |
| 例子 | 重启 Gateway | 010 + watchdog + HEARTBEAT.md 告警规则 |

## 与生物进化的类比

免疫系统不是"修复病毒"，而是"记住病毒的模式，下次更快识别"。

今天的过程就是一次免疫反应：
- 抗原（Gateway 故障）→ 识别（发现心跳滞后）→ 应答（修复 + 分析）→ 记忆细胞（watchdog 自动化 + HEARTBEAT.md 规则）

下次同类故障，10 分钟内自动处理，而不是等 11 小时。**这才是进化，不是优化。**

## 可复用的进化触发条件

1. 故障持续时间 > 1h → 值得写实验笔记
2. 故障涉及架构（非单点配置问题）→ 值得分析脆弱性
3. 修复后能自动化 → 必须自动化，释放注意力给更高层级的思考
4. 一天内从故障到自动化闭环 → 记录为模式，可复制到未来故障

## 对军团进化课题的贡献

这个模式回答了"怎么让军团持续变强"的一个子问题：**不是避免故障，而是建立从故障中学习的系统性机制。**

需要的不是更多 Agent，而是更好的故障→进化管道。
