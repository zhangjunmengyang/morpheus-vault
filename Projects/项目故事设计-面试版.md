## 讲故事原则

### 三层故事结构（每个项目通用）

第一层：背景动机（Why）— 公司/业务背景，痛点

第二层：技术选择（How）— 你做了什么决策？遇到什么困难？为什么选这个方案？这里是区分深度的地方，要能说到底层原理

第三层：结果与洞察（What learned）— 量化结果？系统认知？如果重做会怎么不同？

### 面试高频陷阱与应对

- "你这个和教程有什么区别？" → 强调自己遇到的坑和解法
- "你在里面负责什么？" → 说清楚自己的具体贡献
- "为什么用这个方案不用那个？" → 必须能说出 trade-off

## 项目组合策略

**对于 Agent 应用方向**：主打 P3 + P4，穿插 P5 的后训练结合视角
> "我们在业务 Agent 落地过程中积累了大量真实问题，后来开始思考怎么用 RL 训练 Agent，而不是只靠 prompt 工程。"

**对于后训练方向**：主打 P2，用 P1 作为技术背书，P5 作为应用落地
> "我系统复现了完整后训练 pipeline——从 SFT 到 DPO 再到 GRPO，用 verl 跑过分布式实验，也思考过怎么把后训练用在 Agent 场景。"

**两者都考察时**：P2 → P3 → P5 串成一个故事
> "我对 Agent 系统的理解，从应用层工程一路深入到训练层——因为我看到了 prompt 工程的天花板，觉得真正的解法在模型端。"
