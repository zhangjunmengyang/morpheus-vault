# 一、需求拆解

## 1.1、问题拆解

BA 的需求是一个场景，但本质上是 AI 应用产品**流程问题**，目前整个系统的能力明显受到**流程、框架和模型三方面**的限制。本次主要讨论流程。

目前整个报告流程中面临两方面最大的问题：

### 数据复杂性高

- **数据源问题**：数仓上游的数据数据质量问题，依赖生产方和消费方的常识校验。目前此类问题较少。
- **数据加工**：加工口径问题经常出现，同样依赖生产方和消费方的常识校验，也不好解决。
- **数据消费**： Agent 唯一能做功的点。
- 数据口径问题，这是使用方面临的最大问题，当前 AI 都难以精准判断，因为需要大量的数据知识和业务常识。难以自动检查，依赖大量人工校验。
- 数据工具调用问题，取数不确定性太高，query 的微小变化就会造成数据的偏差，难以自动检查，依赖大量人工校验。
### 分析标准模糊

- **知识收集 & 转化困难**：知识抽取困难，标准无法量化、难以转换为 prompt，分析好坏只能依赖于 BA 的“感觉”。依赖目前的知识甚至无法建立知识库，即使知识库能力具备了，也可能面临整个知识库只有几条知识的窘境，况且知识库的增量知识质量难以保证。
- **后分析流程严重缺失**：目前只会在报告产出前收集知识，但BA 对报告的批注、哪些是好的观点、哪些是冗余的语句的判断信息缺少收集流程。
### 解决方案讨论

- **数据**：能完美保证效果、数据不出错的 Data Agent 方案目前不存在，包括数据探查方案也不明确，没有通用的数据探查方法论，**因此短期靠 Agent 自己走不通**。所以核心观点是，当前阶段就需要重度依赖**人机交互**来解决。
- **分析**：核心能力与**模型能力**并不相关，而是与知识相关，即使后训练提升模型能力还是要依赖知识。本质是通过知识收集效率低、成本高、迁移成本大、认知偏差大、偏好角度复杂、普适性弱，这些**从技术上无法解决**。唯一的解决方式依然是**重流程、重机制，靠人机交互来解决**。
### 核心观点

-  **天才 Agent 并不存在，好的流程胜过一切：**
- 好的流程多 Agent 系统的能力边界 >> 单 Agent 能力边界总和，不好的流程多 Agent 系统的能力边界 << 单 Agent 能力边界总和。
- 同时，流程也不应该是随着迭代而频繁变化的，在整个 Agent / Model 能力升级，新增/去掉一些 Agent/Tool 的时候，不应该影响主体流程，应该具备高度拓展性。
- 分析报告应该边分析边写，即 数据分析 和 summary 不应该是两个步骤，而是二者本身应该发生在同一时刻，真正分析师会做的流程是：**缺什么数据边写边拿，拿不到就不写，写的时候发现不够可以继续拿**。
- **增加异构数据源**：理论上，信息复杂度大幅提升的时候，直到人无法承载的时候，大模型表现出的分析能力越“超出人的预期”。
- 市场环境外部信息源
- 策略、活动、运营动作信息源
- 历史所有真实报告 AI 可见，需要 AI 安全检测/防范 prompt 注入
- **Readfile 不应该与 DA Agent 融合**：所有的工具都不应该与主 agent 融合，核心不在于模型不够强无法掌握，而是plan 和 能力解耦，能简化的是prompt中流程的路由逻辑
- 因此，这引出很重要的一点：主 agent 输出不应该是多种多样的，比如既可以调各种工具输出各样的结果，而是应该一直作为 planner去使用 tool，把所有的能力（包括写作）全部解耦
- 章节 agent 应该作为一个 plan + chat + 调用 tool 的入口。这样才能实现 **指定行为重做 和 chat**，例如区分行级编辑和全章节生成的场景，否则 agent 永远无法作为 chat 入口，永远是一个固化的 sop，详细举例见 1.3 的 report 部分。
- **分段/分块/行级编辑 **报告的能力将是交互上最大的瓶颈，好的产品比如（cursor、原生 claude artifacts）都支持**行级编辑**，对使用效率和稳定性都有很重要的意义。
- **而是应该以机制和流程为核心，而非system prompt**
- MAS 中个人理解最重要的是 context 通信，其中包括记忆的设计，通信的内容。
- 每个 Agent需要什么记忆、需要什么上下文可能都需要进一步探索。

# 二、功能架构

Agent

chat 入口

核心能力

 主 Agent

结束后可以 chat（高级能力）

-  关注全局信息，关注所有章节内容，可以给出一些结合多章节信息的建议
- 预处理工具
- 上传的学城链接解读 + 摘要
- 数据解读 + 摘要 + 用法规划
- 知识库工具
- 历史报告，能预加工抽取与本次节假日相关的数据，提前构建召回 tag，能 get 到某个指标历史的表现，比如去年情人节的 xxx 指标爆发系数，而不仅仅粗粒度召回报告全文
- 知识自动预加工，自行分析本次任务，拿着大量历史报告自问自答，召回，准备先验知识。结合用户提供的新的输入，完成一个预研材料，能做到了解 **历史上这个主题什么指标维度经常看、常规的数据表现怎么样**等。
- 粗粒度规划报告
- 拆分不同章节，大致设计不同层级，写作结构，每个部分希望描述什么内容
- 将粗粒度的交给 章节Agent 来细化
 章节 Agent

- 每个章节的 agent 本质上是一个 agent_node.bind(tool_node)，对话指的是与章节对应的 agent 对话，实现任意工具的重新调用。
生成时和结束后可以 chat

- 关注当前章节的内容，能针对报告内容进行 chat，能通过 chat 随时重新取数、重写章节、行级编辑
- 编辑工具
- 行级编辑、修改
- 能编辑已有报告，如取数口径更新后，可能只一句话，那么其他部分不应该强制被修改。
- 取数工具
- 所有的数据表增加两个信息：① **编号信息 例如 表 2.2.1 **② 生产路径 例如 sql1 --> python code1
- 信息获取工具
- 搜索市场环境外部信息源
- 内部策略、活动、运营动作信息源
- 章节写作工具
- 基于已知信息写作，数据表 + 其他章节关键信息 + 预研材料 + 工具返回信息
- 章节版本
- 每个章节应该具备独立版本，**类似 git log 可以最细以章节粒度看历史修改**
- 报告撰写区域
- 考虑如何能把用户“摘取好的内容”这个关键信息拿到，如果没有这种流程上的机制，靠人来标注是不现实的。
Human node

- 允许任意时刻，干预任何的行为，对标 cursor 的交互自由度。
- 能手动编辑/添加/删除全篇报告中的任意内容，保留用户使用报告的完整轨迹，行为可能指导 Agent：
- **实现高层次人机协同**：“我注意到用户删除了 xxx，并修改了 xxx 为 xxx，我将依照用户的思路...”
- 收集数据
