# Agent 简历描述重点

摘抄自：https://mp.weixin.qq.com/s/vQmS_CxsQwDexQXVysED_Q

**Agent简历描述**侧重在 **多工具（Tool）定义** 与 **工作流（Workflow）设计** 方面的呈现，突出“模型如何决策调用何种工具”的核心逻辑，以便在简历中更好地展示你在 Agent 体系搭建中的技术思路和贡献。

1. **Agent 设计的****四大关键要素**：**工具定义、控制策略、决策逻辑、监控与反馈**；
1. **工作流编排思路**：如何在 **ReAct/Reflection/Memory **等机制下，管理多个工具的调用；
# 一、Agent 设计

1. **工具定义（Tools Definition）**
- 有几个 Agent？分别定位是什么？
- 有哪些工具？分别负责什么？
1. **控制策略（Controller / Orchestrator）**
- 这是 Agent 的“大脑”或“中枢模块”，用于统筹管理对话过程中的**工具调用**。
- 在简历中写明自己设计了一个“Tool Manager”或“Router”，根据对话上下文和用户意图，动态选择或调度对应的工具。
- 此控制策略往往基于**ReAct**或**Plan-Execute**等范式，由模型先产生“思考/计划”（即 Chain of Thought），再根据计划决定是否、何时调用某个工具，并将调用结果反馈给模型进行后续回答。
1. **决策逻辑（Decision Making）**
- **规则驱动**：根据用户输入中出现的关键词、实体类型、或意图分类结果，触发相应工具调用。例如：提及“保费试算”→调起报价工具；提到“最新基金收益”→调起行情搜索。
- **语言模型自发推理（ReAct/Chain-of-Thought）**：让LLM通过内部推理来判断下一步动作。例如：
1. 若模型识别到对话缺乏关键信息，则先询问用户或从 Memory 中检索历史上下文；
1. 若需要外部资料，即调用“知识库检索工具”，获取相关条款文本；
1. 若需检查合规，则再调用“合规审查工具”做最终过滤。
- **多轮决策**：在一段对话中，Agent 可能会多次调用不同工具，先检索、再计算、再审查等——这就需要清晰地定义每个步骤的调用条件和交互结果。
1. **监控与反馈（Monitoring & Feedback）**
- **Memory 记录**：将每一轮调用的上下文信息及工具调用结果存入 Memory，方便后续再用。
- **Reflection / Self-Check**：有些场景下，Agent 需要对自己的回答进行自我评估或审查。如若发现回答不妥或存在合规风险，则再次调用合规工具或修改回复。
- **日志与指标**：在生产环境对各种工具调用进行日志记录，例如：调用次数、耗时、错误率；对对话过程进行A/B测试，量化不同决策链路对结果的影响。
在简历或面试中，你要围绕这些要点突出“**我如何设计了多工具结构、实现了对模型决策的编排**”，让面试官感受到你对Agent体系的深度理解与工程落地能力。

# 二、流程体现

要让LLM“自主”调用工具，核心是一个“**思考-行动-观察-再思考**”的循环。下面给出一个简化版的工作流编排示意：

1. **接收用户请求**：
- 用户输入文本/语音/图片等多模态信息；
- Agent 提取关键要素并进入对话管理流程。
1. **语言模型（LLM）先行思考**：
- 根据对话上下文和用户请求，LLM生成“内部思路”（Chain-of-Thought），这段思路通常包含对问题的理解、缺失信息判断、可能的行动计划等。
- 如：“用户需要保费试算，但我缺少保额和缴费期信息，可先询问用户；或者先调取保险检索工具看看推荐方案。”
1. **LLM发出工具调用指令**：
- 若LLM推理到需要外部信息（如知识库文本、金融数据、计算结果），则发出调用指令（如：CALL TOOL_A with parameters）。
- “Tool Manager”或“Orchestrator”捕捉到指令，调用对应工具执行，获取返回结果。
1. **LLM更新思路并做下一步决策**：
- 接收工具返回结果后，LLM继续Chain-of-Thought。若仍需更多信息或其他工具支持，就继续调用；否则进入回答生成阶段。
- 在某些场景下，如需要合规审查，模型可自动调用**合规审查工具**，或者由系统在回答生成后做二次过滤。
1. **生成最终答案并输出**：
- 将多次工具调用的结果汇总，结合对话上下文进行回答生成。
- 在金融/保险场景下，也可再做一次“Reflection”检测，或提示“以下内容仅供参考，最终以官方条款为准”之类的合规声明。
1. **Memory/Context更新**：
- 将本轮对话、工具调用记录、回答等信息存入Memory，为后续多轮对话提供上下文支持。
在简历中，你可使用类似流程描述来说明你对**LLM多轮对话+工具编排**的理解和实现，特别突出“**模型如何判断何时调用哪个工具**”，例如：

- “自定义了关键词触发 + LLM思考结合的路由策略：当用户输入包含保费计算关键字时，优先调用计算工具；当模型对条款内容不确定时，通过ReAct生成CALL TOOL_A执行检索。”
- “在回答环节，为保证合规性，我在Reflection模块内引入合规审查工具，根据自定义规则集对内容打分并执行拦截或重写。”
# 三、案例

## 多模态保险销售 Agent（着重工具与决策流）

**项目背景：**
 为帮助保险代理人快速检索产品条款、实时获取金融市场数据并执行保费测算，我们构建了一个“多Agent”对话系统，核心在于如何让大语言模型在对话中自行调用**不同工具**（知识库检索、行情查询、报价计算、合规审查等），实现更灵活高效的保险咨询与销售支持。

**我的职责与贡献：**

1. **多工具定义与接口设计**
- 主导设计并实现了 4 大工具：
- **InsuranceKBTool**：对接公司 Milvus 向量库与 BM25 检索系统；
- **FinanceDataTool**：调用第三方金融API，实时获取监管政策、市场指标；
- **PremiumCalculator**：基于产品费率表，执行保费测算及保额计算；
- **ComplianceChecker**：内置合规规则引擎，对输出文本进行审查、打分与阻拦。
- 统一了工具输入输出的数据结构（JSON格式），支持 LLM 在生成时直接插入“调用指令 + 参数”。
1. **ReAct 决策工作流改造**
- 采用 **ReAct（Reason+Act）** 框架，让模型先用“思考”生成对当前对话的推理，再决定“行动”时调用哪一个工具：
- **思考**：LLM 内部判断“是否需要条款详情”“是否需计算保费”或“是否缺少客户年龄信息”；
- **行动**：调用相应工具获取检索结果、费率数据或做合规检测；
- **再思考**：融合工具结果并更新对话策略；
- **回答**：最终向用户输出保险推荐、条款信息或费率测算结论。
- 将**Memory 模块**接入 ReAct 流程，记录每次调用工具的输出及用户反馈，实现跨轮次共享上下文。对涉及客户个人信息的部分做匿名化处理，满足隐私合规。
1. **基于Reflection的合规审查**
- 在最终回答前，模型会调用 **ComplianceChecker** 工具，检查生成内容有无违规字段（如“保证收益”“虚假承诺”）或超监管界限的话术：
- 若审查得分过低，则触发“回答自检与修改”环节，引导模型重写回答；
- 若出现无法挽救的违规话术，系统自动拦截回答并向人工客服或合规岗报警。
- 该流程显著降低了保险销售中“擦边话术”的风险。
1. **关键词触发 + 自发推理结合的决策逻辑**
- 设计了一个“Tool Router”作为后端中间层，监听 LLM 生成的指令。例如：
- 当 LLM 输出“\[CALL:InsuranceKBTool∣query=‘重大疾病险条款’\[CALL:InsuranceKBTool|query=‘重大疾病险条款’]”时，Router 调用相应 RESTful 接口，获取检索结果后再返回给模型。
- 当识别到“保费试算”关键词或 LLM 主动发出“\[CALL:PremiumCalculator∣age=30,product=‘XX险’\[CALL:PremiumCalculator|age=30,product=‘XX险’]”时，Router 便调用报价工具。
- 此外，通过正则或关键词匹配机制，当用户输入包含特定词（如“高收益理财”）时，模型若未自行触发合规审查，也可由系统强制插入合规检查环节，形成“模型推理 + 系统规则”双重保障。
**量化成果：**

- **工具调用准确率**：模型在5000余次真实销售对话中，约 90% 能正确调用相应工具并得到有效结果；对需要多次工具交互（如先检索条款，再计算保费）的复杂问答，也有超过 85% 的成功率。
- **回答准确度与多轮对话提升**：在引入 ReAct+工具后，回答准确率由 80% 提升至 **92%**；用户满意度调查提高至 **4.5/5**；单次对话平均轮数增加了 **30%**（用户更愿意多轮交互获取更具体信息）。
- **合规率**：通过 Reflection+ComplianceChecker 实现对回答内容的实时审查，违规话术出现率由 3% 降至 **0.5%** 以下；由于合规自检能力，减少了约 **60%** 的人工二次复核工作量。
- **代理人使用规模**：系统在 3 个月内覆盖了 **1000+** 代理人日常销售过程，每日调用工具 **2000~3000** 次，极大提升了条款解释和报价效率。
- **性能与响应时延**：借助异步IO与缓存策略，将每次工具调用的平均耗时控制在 **1.8s**；相较初版系统，综合延迟减少 **40%**。
